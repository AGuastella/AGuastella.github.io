Talia

Marketplace monitoring & price-tracking, built for correctness and operability, not volume.

python postgresql llm systems flask

What it does

Talia is a marketplace monitoring and price-tracking platform, currently pointed at second-hand road bikes on Wallapop (Italian and Spanish markets). It runs continuously on a dedicated always-on machine: discovering new listings through the marketplace's search API, tracking every price and status change over time, and scoring candidates against a versioned, buyer-specific rubric using pluggable LLM backends — with a human-in-the-loop review UI producing ground truth to evaluate those models against. It is deliberately personal-scale and rate-limited: one user, one niche, built for correctness and operability rather than volume.

The interesting parts are less "it scrapes a website" and more the engineering around it: strict layering, an upsert-only PostgreSQL design with append-only history, LLM structured output across three interchangeable backends with schema validation and retry, and every non-obvious choice written down in a design decision log.

Architecture

A systemd timer runs the discovery/re-check cycle every 30 minutes:

Discovery (search API) → Parse (pure functions) → PostgreSQL (upsert + history)
   → keyword pre-filter → re-check loop (watching listings) → image download queue

LLM assessment runs out-of-band, never inside the cycle:
   Assessor interface → Anthropic API / Ollama / Gemini → jsonschema validation + retry-then-skip

The database is also read by:
   Flask review UI (screening + ground truth)
   Read-only MCP server (agent access)

Four core modules, each with one narrow responsibility:

ModuleResponsibility
fetch.pyHTTP only — Chrome TLS impersonation; raises typed errors, callers decide what a 404 means
parse.pyPure functions — HTML/JSON in, plain dicts out; no HTTP, no database
db.pyAll persistence — upserts with ON CONFLICT, explicit transactions, single write-path functions
main.pyOrchestration only — the discovery/re-check cycle, subcommands for timers vs. dev loop

The seams are real, not aspirational: parsing is testable against saved fixtures with no network, the database layer is testable against a scratch schema with no scraping, and neither knows the other exists.

Data model

PostgreSQL + pgvector. The design choices that matter:

  • Upsert-only write paths. Every write is INSERT ... ON CONFLICT, never SELECT-then-INSERT. Listings are never deleted; an interest enum (unscreened / watching / ignored) gates which ones the re-check loop spends budget on.
  • Append-only history. price_history and status_history get one row per observed change, not per scrape — the tables answer "when did this drop to €90?" directly.
  • Platform-agnostic without DDL migrations. A platform column, JSONB attributes, and product_types / rubrics tables mean a second marketplace or product category is new rows, not new columns. The code deliberately does not abstract over platforms yet — abstracting from a single example guesses the wrong shape.
  • One write path per table. save_score(), save_extraction(), save_embedding(), save_ground_truth() are the only functions that write their tables, whoever the caller is. Tables storing human annotations are ON DELETE RESTRICT, so no cleanup job can silently eat manual work.
  • Work queues as queries, not tables. Pending image downloads are WHERE downloaded_at IS NULL; the extraction backlog is "watching, and no extraction under the current rubric." Crashed runs resume for free.

LLM assessment

Scoring and attribute extraction run against rubrics stored as data in the database — versioned prompt + JSON schema per rubric — not hardcoded prompts. A small Assessor interface has three interchangeable backends, selected by env var: Anthropic API (structured output via forced tool-use), Ollama (local vision models, structured via the JSON-schema format parameter), and Gemini (plain REST, with a deliberately lossy schema-dialect conversion). All three funnel through one jsonschema validator with a retry-with-errors-then-skip loop — an invalid result can never reach the database, by construction. Every score carries model_id provenance so hosted and local model results coexist in the same table.

Two honest caveats, by design:

  • The pipeline never calls an LLM automatically. Assessment is a separate, budgeted command on its own nightly timer — per-listing API calls across dozens of daily candidates add up, and model quality is being measured before being trusted.
  • Quality is measured against ground truth, not vibes. A local 3B model once confabulated a "seized seatpost" defect straight out of the rubric's own example text — schema-valid, factually wrong. That's exactly why the review UI collects human scores before any backend shootout is run.

Human in the loop

A server-rendered Flask review UI for browsing listings e-shop style: one-tap interest screening, price-drop highlighting, and a ground-truth annotation form (score, defects with severity, even a click-to-measure overlay that estimates frame size from photos using the wheel as a scale reference — human clicks, not a vision model, because VLM keypoints aren't reliable at the tolerance that matters). Model evaluations are collapsed by default on the annotation page to limit anchoring.

Agent access

A read-only MCP server exposes three tools (search listings, listing history, assessment detail) so a coding agent can query the database directly. Read-only is enforced at the connection (default_transaction_read_only = on), not promised in a comment.

Operations

  • Deployment target: an old laptop running bare Linux 24/7. Docker Compose runs Postgres and the review UI continuously; systemd timers fire one-shot containers — the discovery cycle every 30 minutes, the extraction drain nightly.
  • Migration runbook: dev machine → laptop via dump/restore, rehearsed with a dry run against the real data, with the dev database kept as the rollback.
  • Tests (ruff + pytest): thin but honest — they pin decisions, not just behavior. The schema test asserts that deleting an annotated listing is blocked; an exact-string guard on the embedding summary format fails loudly if anyone forks the embedding space.
  • Decision records: the why behind everything non-obvious, including the bugs that shaped the code — e.g. psycopg3's implicit-transaction behavior silently swallowing a whole cycle of writes, and Wallapop's auto-translation silently dropping defect language from a description.
